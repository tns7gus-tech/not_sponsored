# 신뢰도 기반 구매 리서치 에이전트 - 최종 계획서
### (for Codex / Gemini / Claude handoff)

작성일: 2026-03-02  
문서 목적: 이 문서는 사용자의 원래 아이디어, 수정/보완된 방향, 종합 의견, MVP 화면 구조, 검색 로직, 점수 산식, 1차 수익모델, 합법적 개발 가드레일까지 포함한 **실행용 제품 청사진**이다.  
핵심 원칙: 이 서비스는 **"진짜 내돈내산 판결기"** 가 아니라, **"광고/협찬 가능성이 낮고 근거가 투명한 후기들을 우선 탐색해 주는 신뢰도 기반 구매 리서치 에이전트"** 로 설계한다.

---

## 0. 한 줄 결론

이 아이디어는 **방향이 맞다.**  
다만 성공 버전은 `모든 플랫폼의 진짜 일반인 후기 판별기` 가 아니라, 아래처럼 포지셔닝을 바꾼 버전이다.

> **한국형 신뢰도 기반 구매 리서치 에이전트**  
> 사용자가 제품명만 입력하면, 내부적으로 여러 하위 질의를 자동 생성해 여러 소스에 팬아웃 검색을 수행하고,  
> 광고/협찬 가능성이 낮고 근거가 투명한 후기들을 우선 정렬해서 한 번에 보여주는 서비스.

## 1. 원래 아이디어와 보완된 최종 아이디어

### 1.1 사용자의 원래 아이디어
- 사용자는 평소 `나이키 신발 후기`, `아이폰17 내돈내산`, `실사용 리뷰` 같은 검색을 여러 플랫폼에서 각각 반복한다.
- 예: 네이버, 인스타, 스레드, 유튜브, X, 기타 포털/커뮤니티/쇼핑 플랫폼.
- 이를 AI가 대신 수행한다.
- 사용자는 제품명만 입력하면 된다.
- 내부적으로는 아래와 같은 다중 질의가 자동으로 수행된다.
  - `아이폰17`
  - `아이폰17 후기`
  - `아이폰17 리뷰`
  - `아이폰17 내돈내산`
  - `아이폰17 실사용`
  - `아이폰17 장점`
  - `아이폰17 단점`

### 1.2 보완된 최종 아이디어
원래 아이디어의 장점은 강력하지만, 그대로 구현하면 다음 문제가 생긴다.

- `모든 플랫폼` 을 공식적으로 통합 검색하기 어렵다.
- `진짜 일반인 후기만` 을 기계적으로 단정하는 것은 법적/기술적 리스크가 크다.
- 단일 이미지나 단일 문구로 `내돈내산` 을 증명할 수 없다.
- 일부 플랫폼은 공식 API/정책/권한 제약이 크다.

그래서 최종 아이디어는 아래처럼 재정의한다.

### 1.3 최종 제품 정의
**신뢰도 기반 구매 리서치 에이전트**

- 사용자는 제품명 또는 질문만 입력한다.
- 시스템은 내부적으로 검색 의도를 분해하고 다중 질의를 만든다.
- 여러 소스에서 결과를 모은다.
- 각 소스에 대해 `광고성 신호`, `증빙 신호`, `구체성`, `출처 투명성`, `멀티소스 일치도` 를 평가한다.
- 최종적으로 `내돈내산 신뢰도` 혹은 `상업성 주의도` 를 **확률/점수 형태** 로 제시한다.
- 결과는 `단정` 이 아니라 `근거 기반 추정` 으로 보여준다.

## 2. 이 아이디어가 맞는 이유

### 2.1 실제 문제를 해결한다
사용자는 정보가 없는 것이 아니라, 정보가 너무 많고 흩어져 있어서 피곤하다.

현재 구매 전 탐색 행동은 대체로 다음과 같다.
1. 제품명 검색
2. `후기`, `리뷰`, `내돈내산`, `실사용`, `단점` 등을 각각 조합
3. 네이버, 유튜브, 커뮤니티, SNS를 따로 확인
4. 광고 같아 보이는 글을 걸러내기 위해 다시 비교
5. 다시 검색

이 반복을 줄여 주는 것만으로도 큰 가치가 있다.

### 2.2 글로벌 흐름과 맞다
글로벌 대형 플레이어들도 이미 `AI가 구매 리서치를 대신해주는 경험` 으로 움직이고 있다.

- OpenAI는 ChatGPT에 `shopping research` 와 `Instant Checkout` 흐름을 공개했다. (Ref-01, Ref-02)
- Google은 AI Mode에서 `query fan-out` 방식으로 질문을 여러 하위 질의로 분해해 더 깊은 검색을 수행한다고 공식 설명했다. (Ref-03, Ref-04)
- Reddit도 2026년 2월 `쇼핑 검색 경험` 테스트를 공개했다. (Ref-15)

즉, **방향성 자체는 이미 검증된 흐름** 이다.  
기회는 개념 그 자체가 아니라 아래에 있다.

- 한국형 데이터 소스 결합
- 더 간단한 원버튼 UI
- 출처와 근거가 보이는 신뢰도 레이어
- 광고/협찬 가능성에 대한 투명한 설명

## 3. 종합 의견: 긍정적 평가와 부정적 평가

## 3.1 긍정적인 의견

### A. 문제 강도가 높다
사람들은 실제로 제품 구매 전에 여러 플랫폼을 돌아다니며 비교한다.  
특히 한국에서는 네이버 블로그/카페, 유튜브, 쇼핑 리뷰, SNS를 혼합해 보는 행동이 매우 흔하다.

### B. "원버튼" 경험이 강하다
기존 자동화 도구(n8n 등)는 일반인용 제품이 아니라 빌더용 도구에 가깝다.  
반면 이 아이디어는 일반 사용자에게 다음 경험을 준다.

- 제품명만 입력
- 자동 하위 질의 생성
- 다중 소스 검색
- 한 화면 요약
- 근거 카드 제공

이 UX는 강력하다.

### C. 한국 시장에서 시작하기 좋다
네이버 검색 API는 웹/뉴스/블로그/카페글/쇼핑 등 다양한 검색 결과를 웹 서비스나 모바일 앱에 보여줄 수 있도록 안내하고 있고, 검색 API 일일 호출 한도도 공개되어 있다. (Ref-05, Ref-06)

### D. B2C와 B2B 모두 열려 있다
- B2C: 소비자가 더 믿을 수 있는 후기 탐색
- B2B: 브랜드/셀러/리서치 조직이 `광고 제외 실제 VOC` 를 보고 싶어함

---

## 3.2 부정적인 의견

### A. 완전히 새로운 아이디어는 아니다
이미 글로벌하게 다음 방향이 존재한다.

- OpenAI: 쇼핑 리서치/결제 (Ref-01, Ref-02)
- Google: AI Mode + 쇼핑 (Ref-03, Ref-04)
- Reddit: 커뮤니티 기반 쇼핑 검색 테스트 (Ref-15)
- Looria: 신뢰 가능한 리뷰 소스 큐레이션 (Ref-16)

즉, **완전한 독창성은 낮다.**

### B. "모든 플랫폼" 통합은 생각보다 어렵다
공식 API/정책/권한/비용이 플랫폼마다 다르다.

- Threads는 표준 접근에서 일부 Meta 공식 계정 게시물만 조회 가능한 제한이 공식 문서에 있다. (Ref-11)
- Instagram 쪽도 공개 콘텐츠 접근과 해시태그 검색은 제약이 크다. (Ref-12, Ref-13)
- X는 현재 개발자 플랫폼이 pay-per-usage 구조이며 비용 관리가 필요하다. (Ref-10)
- Chrome 확장은 자체 결제가 아니라 웹 결제와 연결하는 구조가 적합하다. (Ref-08)

### C. "진짜 일반인 후기만" 을 보장하면 위험하다
이 제품은 `판결` 이 아니라 `추정/정렬` 도구여야 한다.  
특정 작성자/브랜드/게시물을 `가짜`, `사기`, `조작` 으로 단정하면 명예훼손 및 분쟁 위험이 커진다.

### D. 리뷰 신뢰 서비스 자체만으로 대형 B2C 사업을 만들기 쉽지 않다
Mozilla는 Fakespot을 종료하면서 `아이디어는 공감됐지만 지속 가능한 모델에 맞지 않았다` 고 공식적으로 밝혔다. (Ref-17)  
즉, **기능 가치는 있지만, 단순 리뷰 판별만으로는 수익화가 약할 수 있다.**  
그래서 초기부터 `리서치 에이전트 + 비교 + 저장 + 알림 + B2B VOC` 로 넓게 설계해야 한다.

## 6. 어떤 형태로 시작하는 것이 가장 좋은가

## 6.1 채널 전략 결론
가장 좋은 초기 구조는 아래다.

> **웹 앱 먼저 -> 크롬 확장 추가 -> 모바일 앱/ChatGPT 앱은 나중**

## 6.2 채널별 평가

| 채널 | 역할 | 추천도 | 이유 |
|---|---|---:|---|
| 웹 앱 | 핵심 제품/결제/검색 허브 | 9.0/10 | 가장 빠르고 유연하며 구독/SEO/로그인/결제에 유리 |
| 크롬 확장 | 사용자가 보는 페이지 위 평가/저장 | 8.5/10 | 구매 순간 컨텍스트에서 강력함 |
| 모바일 앱 | 저장/알림/공유/재방문 | 7.0/10 (초기), 8.5/10 (후기) | SNS 사용 맥락에 강하지만 초기 개발비와 운영비 부담 |
| ChatGPT 앱 | 보조 배포 채널 | 5.5/10 (초기), 7.5/10 (후기) | 흥미로운 채널이지만 본체가 되기엔 이름값/생태계 의존 |

## 6.3 "AI agent 앱"에 대한 최종 의견
모바일 앱 자체가 나쁜 것은 아니다.  
하지만 이 서비스의 핵심은 **에이전트 UX** 이지, **모바일 껍데기** 가 아니다.

따라서 초기 MVP는:
- 웹 앱에서 검색과 요약을 먼저 검증
- 이후 크롬 확장으로 구매 순간 개입
- 이후 모바일 앱으로 저장/알림/공유/재방문 강화

가 가장 합리적이다.

## 7. MVP 범위 정의

## 7.1 MVP에서 반드시 해야 하는 것
1. 제품명 입력
2. 자동 하위 질의 생성
3. 여러 소스에서 검색
4. 결과 정규화
5. 광고/협찬/상업성 신호 추출
6. 증빙/구체성/멀티소스 일치도 계산
7. 결과 요약 + 근거 카드 UI
8. 저장/북마크
9. 사용자 피드백 수집

## 7.2 MVP에서 하지 말아야 할 것
- 전 플랫폼 무차별 크롤링
- "진짜 여부" 단정
- 인플루언서/브랜드 블랙리스트
- 감정적 저격 기능
- 대량 원문 저장/재배포
- 자동 로그인 우회/비공개 영역 수집
- 전면 모바일 앱 우선 개발

## 7.3 MVP 추천 소스
### 검색 기반 소스 (공식 접근 우선)
- NAVER Search API (Ref-05, Ref-06)
  - 웹
  - 뉴스
  - 블로그
  - 카페글
  - 쇼핑
- YouTube Data API
  - search.list
  - commentThreads.list 등 (Ref-07)

### 링크 기반 분석 소스 (검색 백본이 아니라 사용자 입력 기반)
- Instagram URL 붙여넣기
- Threads URL 붙여넣기
- X URL 붙여넣기
- 네이버 블로그/카페/기사 URL 붙여넣기
- 쿠팡/스마트스토어/기타 쇼핑 상세 페이지 URL 붙여넣기

### 페이지 컨텍스트 분석
- 크롬 확장에서 사용자가 현재 보고 있는 후기/상품 페이지 분석
- 중앙 서버 크롤링이 아니라 **사용자 측 컨텍스트 분석** 을 우선

## 8. 소스 전략: "검색"과 "링크 분석"을 분리하라

이 프로젝트의 핵심 전략 중 하나는 **모든 플랫폼을 검색엔진처럼 다 뒤지지 않는 것** 이다.

## 8.1 Search Mode
API 친화적인 소스에 대해:
- 검색어 생성
- 결과 수집
- 요약 및 스코어링

예시:
- NAVER 블로그/카페/뉴스/쇼핑
- YouTube
- X(비용 허용 시)

## 8.2 Paste-link Mode
공식적으로 전면 검색이 까다로운 플랫폼은:
- 사용자가 링크를 붙여넣거나 공유
- 시스템이 해당 URL만 분석

예시:
- Instagram
- Threads
- 일부 쇼핑 리뷰 페이지
- 특정 블로그/기사

## 8.3 Page-overlay Mode
사용자가 이미 보고 있는 페이지 위에서:
- 광고성 신호
- 제휴 링크
- 상업성 문구
- 반복 키워드
- 저장/북마크/비교
를 제공

예시:
- 네이버 블로그 글 위 오버레이
- 쇼핑 상세 페이지 옆 사이드 패널
- 유튜브 리뷰 페이지 요약

이 구조가 중요한 이유:
- 법적 리스크 감소
- 플랫폼 정책 충돌 감소
- 초기 개발 속도 증가

## 11. 점수 산식 설계

## 11.1 핵심 원칙
- 점수는 `사실 확정` 이 아니라 `추정치`
- 점수만 보여주지 말고 이유를 함께 설명
- 사용자에게는 이해 가능한 언어로 표현
- 내부 점수와 사용자 노출 점수를 분리 가능

## 11.2 내부 점수 4종
1. **Commercial Risk Score (CRS)**  
   상업성/광고성/협찬 가능성

2. **Evidence Quality Score (EQS)**  
   구매/사용/비교/구체성 근거의 질

3. **Source Credibility Score (SCS)**  
   출처 신뢰성과 계정/채널의 일반성

4. **Consensus Score (COS)**  
   여러 소스에서 비슷한 장점/단점이 반복되는 정도

## 11.3 예시 피처 목록

### Commercial Risk 관련 피처
- `#광고`, `#협찬`, `#파트너스`, `제공받아`, `브랜드로부터`
- 제휴 링크 포함 여부
- 쿠폰코드/공구코드 포함 여부
- CTA 강도 (`지금 구매`, `링크는 댓글`, `무조건 사세요`)
- 특정 브랜드 편중 게시 패턴
- 게시물마다 유사 문장 반복

### Evidence Quality 관련 피처
- 1인칭 사용 경험 (`내가 써봤는데`, `3주 사용`, `두 달 사용`)
- 구매 이유/상황
- 구체적 장단점
- 비교 대상 존재
- 사용 기간 언급
- 환불/교환/AS 경험
- 사진/영상/실물 묘사

### Source Credibility 관련 피처
- 일반 사용자 계정 패턴
- 특정 브랜드 집중도 낮음
- 게시물 카테고리 다양성
- 과도한 상업문구 비율 낮음
- 원문 링크/출처명확성

### Consensus 관련 피처
- 멀티소스에서 반복되는 장점
- 멀티소스에서 반복되는 단점
- 특정 이슈에 대한 상호 확인
- 특정 홍보 문구의 비정상적 반복 여부

## 11.4 예시 산식 (0~1 정규화)

```text
CRS = 0.30*disclosure_signal
    + 0.20*affiliate_signal
    + 0.15*coupon_or_groupbuy_signal
    + 0.15*cta_intensity
    + 0.10*brand_concentration
    + 0.10*template_repetition

EQS = 0.20*first_person_usage
    + 0.20*usage_duration
    + 0.15*specific_pros_cons
    + 0.10*purchase_context
    + 0.10*after_service_or_refund_context
    + 0.10*comparison_depth
    + 0.15*media_evidence

SCS = 0.30*account_normality
    + 0.20*source_transparency
    + 0.20*content_diversity
    + 0.15*historical_non_commercial_ratio
    + 0.15*author_consistency

COS = 0.40*pros_agreement
    + 0.40*cons_agreement
    + 0.20*issue_confirmation
```

## 11.5 사용자 노출 점수
사용자에게 보여줄 점수는 아래처럼 단순화한다.

```text
Authenticity Confidence (AC)
= clamp(
    100 * (
      0.35*EQS
    + 0.25*SCS
    + 0.20*COS
    + 0.10*freshness
    + 0.10*diversity
    - 0.35*CRS
    - 0.05*manipulation_risk
    )
)
```

## 11.6 사용자-facing 레이블
- **80~100**: 신뢰도 높음
- **60~79**: 참고 가능
- **40~59**: 혼합/주의
- **0~39**: 상업성 또는 근거 부족 주의

중요:
- `0~39` 도 `가짜` 라고 부르지 않는다.
- `상업성 신호가 높거나 근거가 충분하지 않음` 으로 표시한다.

## 11.7 설명 가능한 결과 예시
- 신뢰도: 78/100
- 이유:
  - 3주 사용 경험이 구체적으로 서술됨
  - 장점/단점이 구체적
  - 링크/쿠폰/공구 코드 없음
  - 다른 5개 출처에서도 비슷한 단점이 반복됨
  - 단, 실결제 증빙은 없음

## 14.4 허용 가능한 후속 수익화
### Opt-in 제휴
Chrome 확장과 관련해서는 Chrome 정책상 제휴 프로그램은 명확히 고지되어야 하고, 사용자에게 직접적이고 투명한 이익이 있을 때만 허용된다. (Ref-09)

따라서 제휴를 붙인다면 반드시:
- 기본 OFF
- 명시적 동의
- 사용자 할인/혜택 존재
- 점수 랭킹과 분리
- `추천 랭킹` 과 `제휴 상거래` 를 화면상 분리

## 15. 합법적으로 개발하기 위한 가드레일

> 아래 내용은 제품/컴플라이언스 설계 원칙이며, 정식 출시 전에는 한국 변호사 자문 1회를 받는 것을 강력 권장한다.

## 15.1 절대 단정하지 말 것
- `이 글은 가짜다`
- `이 사람은 조작했다`
- `이 브랜드는 사기다`

이런 표현은 피한다.

대신:
- `광고/협찬 가능성이 높음`
- `근거가 충분하지 않음`
- `상업성 신호가 탐지됨`
- `자동 추정 결과이며 사실 확정이 아님`

## 15.2 명예훼손/권리침해 리스크 줄이기
한국 정보통신망법에는 타인의 권리를 침해하는 정보 유통 금지와 권리보호 관련 조항이 있고, 명예훼손 관련 분쟁 및 판례도 존재한다. (Ref-20, Ref-21)

제품 정책:
- 사람/브랜드를 공개 블랙리스트화 하지 않는다.
- 사용자 댓글/평가도 모더레이션한다.
- 신고/정정/삭제 요청 프로세스를 만든다.
- 결과 화면에 면책 문구를 둔다.
- "AI 추정" 과 "원문 근거" 를 분리해서 보여준다.

## 15.3 표시광고/추천보증 관련 고려
한국에서는 표시·광고의 공정화에 관한 법률이 존재하고, 공정거래위원회는 추천·보증과 경제적 이해관계 공개 관련 가이드를 제공하고 있다. (Ref-18, Ref-19)

활용 방향:
- 이 법/지침은 당신 서비스가 `광고/협찬 문구` 를 찾는 기준을 설계할 때 참고할 수 있다.
- 단, 이 지침을 근거로 개별 게시물의 위법 여부를 단정하지는 않는다.
- "공개 원칙에 비추어 상업성 신호가 있는지" 정도만 신호로 활용한다.

## 15.4 개인정보 최소 수집
개인정보보호법상 개인정보 처리 목적, 수집/이용 근거, 처리방침 공개 등이 중요하다. (Ref-22, Ref-23, Ref-24)

제품 정책:
- 주소, 전화번호, 주문번호 같은 민감 필드는 자동 마스킹
- 원문 전체 저장보다 링크 + 최소 발췌 + 파생 요약 중심
- 서버 전송은 최소화
- 링크 분석 업로드 시 이용 목적과 보관 기간 명시
- 개인정보 처리방침 공개
- 삭제 요청 경로 제공

## 15.5 저작권/재배포 최소화
권장 원칙:
- 원문 미러링 금지
- 전체 게시물 복제 금지
- 작은 발췌 + 링크 + 요약/분석 중심
- 이미지는 되도록 원문 링크로 보내고, 썸네일/미리보기는 최소화
- 캐시는 단기/필수 범위로 제한

## 15.6 크롤링/약관 리스크 최소화
원칙:
- 공식 API 우선
- 사용자 주도 URL 분석 우선
- 사용자가 보는 페이지에서의 클라이언트 측 보조 기능 우선
- 자동 로그인 우회 금지
- 대량 수집/대량 재배포 금지
- robots/약관/플랫폼 정책을 반드시 검토

## 15.7 크롬 확장 관련 주의
- Chrome Web Store 자체 결제 시스템은 deprecated 상태다. (Ref-08)
- 확장은 결제 허브가 아니라 `전환 도구` 로 보는 것이 맞다.
- 제휴 프로그램이 있다면 설치 전과 UI 내에 명확히 고지해야 한다. (Ref-09)
- 관련 사용자 행동 없이 제휴 링크를 삽입하면 안 된다. (Ref-09)

## 23. 이 아이디어가 이길 수 있는 이유

경쟁 제품이나 대형 플레이어가 이미 있다는 사실은 오히려 좋은 신호다.  
이 아이디어가 이길 수 있는 이유는 아래와 같다.

### A. 한국형 데이터 우선
- 네이버 중심 행동 이해
- 한국어 후기 패턴 이해
- 블로그/카페/실사용 맥락 반영

### B. 신뢰도 레이어
단순 요약이 아니라:
- 상업성 신호
- 근거 신호
- 멀티소스 일치도
를 함께 보여준다.

### C. 원버튼 UX
복잡한 자동화가 아니라:
- 제품명 입력
- 자동 팬아웃 검색
- 한 화면 요약

### D. 검색 + 링크 + 페이지 분석 결합
다른 제품보다 현실적인 접근이다.
- 검색 가능한 곳은 검색
- 어려운 곳은 링크 분석
- 구매 순간에는 확장 프로그램 개입
